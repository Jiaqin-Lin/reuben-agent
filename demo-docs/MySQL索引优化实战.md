# MySQL 索引优化实战

## 索引基础

索引是数据库管理系统中的一个排序数据结构，用于快速查询和检索数据。没有索引时，数据库必须执行全表扫描（Full Table Scan）来查找匹配的行，时间复杂度为 O(n)。有了合适的索引，查询时间复杂度可以降低到 O(log n) 甚至 O(1)。

MySQL 中最常用的索引实现是 B+Tree。B+Tree 是一棵平衡多路搜索树，所有数据都存储在叶子节点中，叶子节点通过双向链表连接。这使得 B+Tree 既能高效支持等值查询，也能高效支持范围查询。

## B+Tree 索引原理

B+Tree 的关键特性包括：
- 所有叶子节点在同一层，保证了查询时间的稳定性
- 非叶子节点只存储索引键，不存储数据，使得单个节点能容纳更多键，降低树的高度
- 叶子节点通过指针串联成有序链表，支持高效的范围扫描（Range Scan）
- 通常一个 3-4 层的 B+Tree 就能索引上千万行数据

InnoDB 存储引擎中，B+Tree 节点的大小等于一个数据页（默认 16KB）。当数据页填满时，节点分裂为两个节点，各自包含约一半的数据。这一过程由 InnoDB 自动管理，对应用层透明。

## 聚簇索引与二级索引

InnoDB 使用聚簇索引（Clustered Index）组织数据。聚簇索引的叶子节点直接存储完整的行数据。每张表只能有一个聚簇索引——如果定义了主键，InnoDB 使用主键作为聚簇索引；如果没有主键，InnoDB 选择第一个唯一非空索引；如果都没有，InnoDB 自动生成一个隐藏的 6 字节 ROW ID。

二级索引（Secondary Index）的叶子节点存储的是对应的聚簇索引键值。通过二级索引查询时，需要先查二级索引获取主键值，再通过主键回表查询完整行数据——这个过程称为回表。当查询的列都在二级索引中时，可以避免回表，称为覆盖索引。

## 索引优化策略

### 最左前缀原则

联合索引遵循最左前缀匹配规则。例如创建联合索引 (a, b, c)，以下查询可以利用索引：
- WHERE a = 1（使用 a）
- WHERE a = 1 AND b = 2（使用 a, b）
- WHERE a = 1 AND b = 2 AND c = 3（使用全部三列）
- WHERE a = 1 AND c = 3（使用 a 过滤，c 无法使用索引）

以下查询无法使用索引：
- WHERE b = 2（跳过了 a，不满足最左前缀）
- WHERE c = 3（跳过了 a 和 b）

### 索引下推

MySQL 5.6 引入索引下推特性。当查询使用联合索引但部分条件无法使用索引时，MySQL 将无法使用索引的条件推送到存储引擎层进行过滤，减少回表次数。

### 避免索引失效

以下写法会导致索引失效：
- WHERE 子句中对索引列使用函数（如 WHERE YEAR(create_time) = 2024）
- WHERE 子句中对索引列进行运算（如 WHERE id + 1 = 100）
- 隐式类型转换（如 varchar 列与整数比较）
- LIKE 以通配符开头（LIKE '%keyword'），但 LIKE 'keyword%' 可以使用索引
- OR 条件中不全是索引列

## 慢查询分析

### EXPLAIN 解读

EXPLAIN 是 MySQL 中分析查询执行计划的重要工具。关键字段包括：
- type：访问类型，从优到劣依次为 system > const > eq_ref > ref > range > index > ALL
- key：实际使用的索引
- rows：MySQL 预估需要扫描的行数
- Extra：额外信息，Using index 表示覆盖索引（无需回表），Using filesort 表示需要额外排序（可能需要优化），Using temporary 表示使用临时表（可能影响性能）

### 慢查询日志

通过 slow_query_log 参数开启慢查询日志，并设置 long_query_time 阈值（如 200ms）。定期分析慢查询日志，发现需要优化的 SQL。pt-query-digest 是 Percona Toolkit 中的慢查询分析工具，可以提供详细的统计报告。

## 索引维护

### 索引碎片整理

随着数据的增删改，索引会产生碎片，影响查询性能和空间利用率。通过 OPTIMIZE TABLE 或 ALTER TABLE ENGINE=InnoDB 可以重建表和索引，回收空间并整理碎片。在 MySQL 8.0 中，支持在线 DDL，重建索引不会阻塞读写操作。

### 冗余索引检测

冗余索引不仅浪费存储空间，还会降低写入性能（因为每次 INSERT、UPDATE、DELETE 都需要维护所有相关索引）。MySQL 8.0 的 sys schema 提供了 schema_redundant_indexes 视图，可以自动检测冗余索引。

例如，如果已经存在联合索引 (a, b)，那么单独的索引 (a) 就是冗余的——因为最左前缀原则保证了查询 a 列时可以使用联合索引。

### 索引监控

通过 Performance Schema 可以监控索引的使用情况。sys schema 中的 schema_unused_indexes 视图可以找出从未被使用的索引，这些是删除的候选对象。但在删除前需要确认这些索引是否用于约束检查或其他特殊场景。

## 实战案例

### 分页优化

深分页是一个常见问题。如 LIMIT 1000000, 20 需要扫描并丢弃前 100 万行，效率极低。优化方法包括：
- 使用游标分页（WHERE id > last_max_id LIMIT 20）
- 使用子查询先获取主键列表再关联回表
- 避免在大表上使用 OFFSET 过大的分页

### 排序优化

当 ORDER BY 的列包含在索引中时，MySQL 可以直接通过索引返回有序结果，避免额外的排序操作。创建联合索引时，将等值查询列放在前面，排序查询列放在后面，可以最大化利用索引。

### 关联查询优化

多表 JOIN 查询时，MySQL 通常选择驱动表全表扫描，被驱动表走索引。因此需要确保 JOIN 条件列上有索引。使用 STRAIGHT_JOIN 可以强制指定驱动表顺序，覆盖 MySQL 优化器可能做出的次优选择。
